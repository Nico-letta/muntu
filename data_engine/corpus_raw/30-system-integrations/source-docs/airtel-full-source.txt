OAuth2
This API is used to get the bearer token. The output of this API contains access_token that will be used as bearer token for the API that we will be going to call.

POST /auth/oauth2/token

Note : This is a common API to fetch access token for all the APIs listed in the documentation section.

Callback Encryption
Partner Can Authenticate the response sent to them regarding the status of the transaction on the callback url. It can be configured in Application settings. Along with the callback request body a , hash message will be sent to user. User can hash the the callback request with the private key shown in application setting using HmacSHA256 algorithm, with output text format in Base64 and can match with it hash message sent if Callback authentication is enabled.

SHA256 with RSA signature is an efficient asymmetric encryption method used in many secure APIs. This algorithm first calculates a unique hash of the input data using SHA256 algorithm. The hash is then encrypted with a private key using the RSA algorithm. Steps to enable callback authentication

Step1 : Go to application settings

Step 2: Enable callback authentication

Step 3: Copy Private key to encrypt using HMAC algortihm

Note : Private key is available in application settings of account . Copy it from there.

Message Signing
Message signing in requests is a security technique to ensure the authenticity and integrity of data sent between parties. It involves generating a key , creating a payload, calculating a hash of the payload, signing the hash with the generated key, and including the signature in the request. On the server side, the private key is used to verify the signature, ensuring the data hasn't been tampered with. This process safeguards against data manipulation and impersonation during transmission. We use a similar technique to sign our request. Partner needs to encrypt the payload using a randomly generated AES key iv pair . Encrypt the key iv using a RSA key and add both encrypted key and encrypted signature to the request as headers. At the server side we use private RSA key to decrypt the AES key iv and encrypt the payload with this key and iv. We match this generated encrypted payload to the signature received to ensure that there is no change in the payload sent by the partner and payload received.

STEPS :

1. Generate a random AES key of size 256 bits and iv of size 128 bits using AES/CBC/PKCS5Padding algorithm.

2. Generated AES key and iv should be base 64 encoded.
Example - Key(PSw37xtnShLl7zgWn4dLSnf1J5GRhRsOD4OfvJOuLIM=) and iv(gGIhDvCBnSBhBgYiXCyMnw==)

              OverDraft Loans
Overdraft loan is a type of loan that is provided to a consumer (agent / subscriber) when they are making a financial transaction but do not have sufficient balance in their respective wallet(s).

Base URLs:

Staging -- https://openapiuat.airtel.cg

Production -- https://openapi.airtel.cg

Repay Loan Callback
This API is to be used by partners to notify Loan Switch the success/failure of the loan repayment request

POST /overdraft-loans/v1/callback/repayment

Parameters
Name	In	Type	Required	Description
Content-Type	header	string	true	Specifies the media type of the request
Accept	header	string	true	Specifies the media type of the response
x-country	header	string	true	Country Code
Authorization	header	string	true	Authorization bearer token
Request Body
Attribute	Description
st
boolean
mandatory	Status Flag. st will be "true" in case of processing status "PROCESSED/QUEUED" and "false" in case of "REJECTED"
txn_res
object
mandatory	Transaction response
txn_res.processing_status
string
mandatory	Status of the transaction. It can be PROCESSED / QUEUED / REJECTED
txn_res.txn_id
string
mandatory	Unique transaction id of Airtel loan switch
txn_res.auth_code
string
mandatory	Partner Transaction Id
txn_res.partner_response
object
mandatory	Partner response object
partner_response.msg
string
mandatory	Success or Failure message
partner_response.total_amt_paid
number
mandatory	Total Paid amount (Principle amount paid + Fees Amount Paid)
partner_response.principle_amt_paid
number
mandatory	Total Principle Amount Paid
partner_response.fees_amt_paid
number
mandatory	Total Fees amount Paid
partner_response.user_repay_loan_details
array
mandatory	Loan repayment details array. Contains details of all existing loans of the user
user_repay_loan_details.loan_id
string
mandatory	Partner Loan ID
user_repay_loan_details.total_amt_paid
number
mandatory	Total loan amount repaid for the current transaction
user_repay_loan_details.principle_amt_paid
number
mandatory	Amount deducted from principle loan amount
user_repay_loan_details.fees_amt_paid
number
mandatory	Amount deducted from loan fee amount
user_repay_loan_details.total_outstanding_amt
number
mandatory	Total Outstanding Amount after repayment
user_repay_loan_details.is_cleared
boolean
mandatory	Loan is Cleared or not
Response Body
Attribute	Description
status
object
Status
status.response_code
string
Product specific code to identify the error and success response. This will be different for the type of error and success.
status.code
string
Status code, HTTP status code
status.success
boolean
Status of API, True: if API is run successfully, False: if there is any failure (validation, exception, etc)
status.message
string
The descriptive message of the response/action (Success, Failure etc)

          RemX
Overview
The RemX suite enables seamless electronic fund transfers across borders, empowering individuals and organizations to send money internationally. It connects various financial institutions and payment service providers, streamlining the remittance process—especially for transactions from non-Airtel ecosystems to Airtel subscribers.

Product Architecture
Flow: Kong Gateway → MFS Platform (API Layer) → MQ (Transaction Processing)

Partners & Use Cases
Currently, both products support transactions from non-Airtel partners (e.g., Thunes, TerraPay) to Airtel subscribers only. Partners can integrate with the Developer Portal APIs to facilitate international remittances to Airtel customers.

Credit: Transfers an amount from a Payer account to a payee account.
Validate: Verifies if a customer is active and able to receive funds.
Refund: Makes a full refund to partners.
Status: Checks the status of the transaction.
Integration & Validation Process
Authorization & Token Generation
Authenticate using your client_id and client_secret to generate an Authorization Token.
API Used: Token Generation (OAuth)
Note: The token is valid for 180 seconds and must be refreshed upon expiry.

KYC (Know Your Customer) Verification
Validate the payee’s KYC details before transferring funds.
API Used: Validate API
Purpose: Ensures the recipient’s wallet is legitimate and active, reducing fraud risk.

Initiating the Transfer
Once the payee is verified and the balance is sufficient, initiate the transfer.
API Used: Credit API
Details Required: Payer and payee wallet information, transfer amount, and related data.

Response Handling
Process the system’s response to the transfer request, which may indicate success, failure, in progress, or an ambiguous state.

Transaction Enquiry
Query the status of a transaction after initiation to check completion or troubleshoot issues.
API Used: Status API

Refunds
Initiate a full refund if required.
API Used: Refund API
Error Handling and Categorization
The Remittance APIs facilitate the electronic transfer of funds across financial institutions and payment service providers, handling international remittances from non-Airtel partners to Airtel subscribers. Throughout the transaction process, robust error handling ensures that both partners and end users are informed of issues promptly and accurately.

Remittance transactions typically involve deducting funds from a partner’s account and crediting the recipient’s Airtel wallet. The process includes multiple steps—such as KYC validation, balance checks, and transfer initiation—each of which may encounter specific error scenarios.

If any issues arise at any stage, the APIs return standardized error codes and messages to indicate the cause of failure. Common error situations include:

Insufficient funds in the partner’s account
Invalid or incomplete KYC details for the recipient
Issues with recipient wallet
Each API response includes an error code and a descriptive message to help partners identify and resolve issues quickly. A list of error codes and their meanings is available in the API documentation, allowing partners to implement precise and user-friendly error handling within their platforms.

By following the prescribed integration steps and handling API responses as documented, partners can ensure seamless and secure remittance experiences for their users.

           RemX
Overview
The RemX suite enables seamless electronic fund transfers across borders, empowering individuals and organizations to send money internationally. It connects various financial institutions and payment service providers, streamlining the remittance process—especially for transactions from non-Airtel ecosystems to Airtel subscribers.

Product Architecture
Flow: Kong Gateway → MFS Platform (API Layer) → MQ (Transaction Processing)

Partners & Use Cases
Currently, both products support transactions from non-Airtel partners (e.g., Thunes, TerraPay) to Airtel subscribers only. Partners can integrate with the Developer Portal APIs to facilitate international remittances to Airtel customers.

Credit: Transfers an amount from a Payer account to a payee account.
Validate: Verifies if a customer is active and able to receive funds.
Refund: Makes a full refund to partners.
Status: Checks the status of the transaction.
Integration & Validation Process
Authorization & Token Generation
Authenticate using your client_id and client_secret to generate an Authorization Token.
API Used: Token Generation (OAuth)
Note: The token is valid for 180 seconds and must be refreshed upon expiry.

KYC (Know Your Customer) Verification
Validate the payee’s KYC details before transferring funds.
API Used: Validate API
Purpose: Ensures the recipient’s wallet is legitimate and active, reducing fraud risk.

Initiating the Transfer
Once the payee is verified and the balance is sufficient, initiate the transfer.
API Used: Credit API
Details Required: Payer and payee wallet information, transfer amount, and related data.

Response Handling
Process the system’s response to the transfer request, which may indicate success, failure, in progress, or an ambiguous state.

Transaction Enquiry
Query the status of a transaction after initiation to check completion or troubleshoot issues.
API Used: Status API

Refunds
Initiate a full refund if required.
API Used: Refund API
Error Handling and Categorization
The Remittance APIs facilitate the electronic transfer of funds across financial institutions and payment service providers, handling international remittances from non-Airtel partners to Airtel subscribers. Throughout the transaction process, robust error handling ensures that both partners and end users are informed of issues promptly and accurately.

Remittance transactions typically involve deducting funds from a partner’s account and crediting the recipient’s Airtel wallet. The process includes multiple steps—such as KYC validation, balance checks, and transfer initiation—each of which may encounter specific error scenarios.

If any issues arise at any stage, the APIs return standardized error codes and messages to indicate the cause of failure. Common error situations include:

Insufficient funds in the partner’s account
Invalid or incomplete KYC details for the recipient
Issues with recipient wallet
Each API response includes an error code and a descriptive message to help partners identify and resolve issues quickly. A list of error codes and their meanings is available in the API documentation, allowing partners to implement precise and user-friendly error handling within their platforms.

By following the prescribed integration steps and handling API responses as documented, partners can ensure seamless and secure remittance experiences for their users.

                 RemX
Overview
The RemX suite enables seamless electronic fund transfers across borders, empowering individuals and organizations to send money internationally. It connects various financial institutions and payment service providers, streamlining the remittance process—especially for transactions from non-Airtel ecosystems to Airtel subscribers.

Product Architecture
Flow: Kong Gateway → MFS Platform (API Layer) → MQ (Transaction Processing)

Partners & Use Cases
Currently, both products support transactions from non-Airtel partners (e.g., Thunes, TerraPay) to Airtel subscribers only. Partners can integrate with the Developer Portal APIs to facilitate international remittances to Airtel customers.

Credit: Transfers an amount from a Payer account to a payee account.
Validate: Verifies if a customer is active and able to receive funds.
Refund: Makes a full refund to partners.
Status: Checks the status of the transaction.
Integration & Validation Process
Authorization & Token Generation
Authenticate using your client_id and client_secret to generate an Authorization Token.
API Used: Token Generation (OAuth)
Note: The token is valid for 180 seconds and must be refreshed upon expiry.

KYC (Know Your Customer) Verification
Validate the payee’s KYC details before transferring funds.
API Used: Validate API
Purpose: Ensures the recipient’s wallet is legitimate and active, reducing fraud risk.

Initiating the Transfer
Once the payee is verified and the balance is sufficient, initiate the transfer.
API Used: Credit API
Details Required: Payer and payee wallet information, transfer amount, and related data.

Response Handling
Process the system’s response to the transfer request, which may indicate success, failure, in progress, or an ambiguous state.

Transaction Enquiry
Query the status of a transaction after initiation to check completion or troubleshoot issues.
API Used: Status API

Refunds
Initiate a full refund if required.
API Used: Refund API
Error Handling and Categorization
The Remittance APIs facilitate the electronic transfer of funds across financial institutions and payment service providers, handling international remittances from non-Airtel partners to Airtel subscribers. Throughout the transaction process, robust error handling ensures that both partners and end users are informed of issues promptly and accurately.

Remittance transactions typically involve deducting funds from a partner’s account and crediting the recipient’s Airtel wallet. The process includes multiple steps—such as KYC validation, balance checks, and transfer initiation—each of which may encounter specific error scenarios.

If any issues arise at any stage, the APIs return standardized error codes and messages to indicate the cause of failure. Common error situations include:

Insufficient funds in the partner’s account
Invalid or incomplete KYC details for the recipient
Issues with recipient wallet
Each API response includes an error code and a descriptive message to help partners identify and resolve issues quickly. A list of error codes and their meanings is available in the API documentation, allowing partners to implement precise and user-friendly error handling within their platforms.

By following the prescribed integration steps and handling API responses as documented, partners can ensure seamless and secure remittance experiences for their users.

           RemX
Overview
The RemX suite enables seamless electronic fund transfers across borders, empowering individuals and organizations to send money internationally. It connects various financial institutions and payment service providers, streamlining the remittance process—especially for transactions from non-Airtel ecosystems to Airtel subscribers.

Product Architecture
Flow: Kong Gateway → MFS Platform (API Layer) → MQ (Transaction Processing)

Partners & Use Cases
Currently, both products support transactions from non-Airtel partners (e.g., Thunes, TerraPay) to Airtel subscribers only. Partners can integrate with the Developer Portal APIs to facilitate international remittances to Airtel customers.

Credit: Transfers an amount from a Payer account to a payee account.
Validate: Verifies if a customer is active and able to receive funds.
Refund: Makes a full refund to partners.
Status: Checks the status of the transaction.
Integration & Validation Process
Authorization & Token Generation
Authenticate using your client_id and client_secret to generate an Authorization Token.
API Used: Token Generation (OAuth)
Note: The token is valid for 180 seconds and must be refreshed upon expiry.

KYC (Know Your Customer) Verification
Validate the payee’s KYC details before transferring funds.
API Used: Validate API
Purpose: Ensures the recipient’s wallet is legitimate and active, reducing fraud risk.

Initiating the Transfer
Once the payee is verified and the balance is sufficient, initiate the transfer.
API Used: Credit API
Details Required: Payer and payee wallet information, transfer amount, and related data.

Response Handling
Process the system’s response to the transfer request, which may indicate success, failure, in progress, or an ambiguous state.

Transaction Enquiry
Query the status of a transaction after initiation to check completion or troubleshoot issues.
API Used: Status API

Refunds
Initiate a full refund if required.
API Used: Refund API
Error Handling and Categorization
The Remittance APIs facilitate the electronic transfer of funds across financial institutions and payment service providers, handling international remittances from non-Airtel partners to Airtel subscribers. Throughout the transaction process, robust error handling ensures that both partners and end users are informed of issues promptly and accurately.

Remittance transactions typically involve deducting funds from a partner’s account and crediting the recipient’s Airtel wallet. The process includes multiple steps—such as KYC validation, balance checks, and transfer initiation—each of which may encounter specific error scenarios.

If any issues arise at any stage, the APIs return standardized error codes and messages to indicate the cause of failure. Common error situations include:

Insufficient funds in the partner’s account
Invalid or incomplete KYC details for the recipient
Issues with recipient wallet
Each API response includes an error code and a descriptive message to help partners identify and resolve issues quickly. A list of error codes and their meanings is available in the API documentation, allowing partners to implement precise and user-friendly error handling within their platforms.

By following the prescribed integration steps and handling API responses as documented, partners can ensure seamless and secure remittance experiences for their users.
